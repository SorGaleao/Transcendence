# 🧠 Git 101 (Team Workflow Edition)

## 1. Core Idea 

* **`main` (or `master`) = stable, production-ready code**
* You **never work directly on main**
* Instead: create branches → make changes → merge back

---

# 🌿 2. Branching Basics

## Creating a branch

```bash
git checkout -b feature/login-page
```

This:

* creates a new branch
* switches you to it

## Naming branches (best practices)

Use **clear, structured names**:

```
feature/<what-you-built>
bugfix/<what-you-fixed>
hotfix/<urgent-fix>
chore/<maintenance>
```

Examples:

* `feature/user-auth`
* `bugfix/navbar-crash`
* `feature/api-integration`

👉 Keep names:

* lowercase
* hyphen-separated
* descriptive but short

---

# 🔄 3. Typical Workflow (daily use)

### 1. Get latest code

```bash
git checkout main
git pull
```

### 2. Create branch

```bash
git checkout -b feature/your-task
```

### 3. Work + commit

```bash
git add .
git commit -m "Add login validation"
```

### 4. Push branch

```bash
git push origin feature/your-task
```

### 5. Open Pull Request (PR)

* teammates review your code
* discuss changes
* then merge into `main`

---

# 👥 4. Can multiple people edit the same file?

### YES — but here's what happens:

## Case A: Different parts of file

Git merges automatically ✅

## Case B: Same lines edited

Git creates a **merge conflict** ❗

Example conflict:

```text
<<<<<<< HEAD
your code
=======
their code
>>>>>>> branch-name
```

You must manually:

* choose one
* or combine both

Then:

```bash
git add .
git commit
```

👉 So yes, collaboration works—but conflicts need human resolution.

# 🔀 5. Merge vs Rebase (simple explanation)

## Merge (default & safest)

```bash
git merge main
```

* keeps full history
* adds a "merge commit"
* easier for beginners

👉 Think: *"combine timelines"*

---

## Rebase (cleaner history)

```bash
git rebase main
```

* moves your branch on top of latest main
* creates a **linear history**
* avoids messy merge commits

👉 Think: *"pretend you started from latest version"*

---

## ⚠️ Important rule

* Use **rebase for your own branches**
* Avoid rebasing shared branches (can mess up teammates)

---

# 🔁 6. Keeping your branch up to date

While working, `main` changes.

### Option A (safe):

```bash
git merge main
```

### Option B (cleaner):

```bash
git rebase main
```

# 🧹 7. Clean commit habits

Bad:

```
fix stuff
update
changes
```

Good:

```
Add login validation
Fix navbar overflow bug
Refactor API error handling
```

👉 Each commit should:

* do ONE logical thing
* be understandable later

---

# 🚫 8. What NOT to do in team projects

❌ Push directly to `main`
❌ Work on outdated branches
❌ Make huge commits ("all changes")
❌ Ignore merge conflicts
❌ Rewrite shared history (dangerous)

---

# 🧭 9. Simple Team Workflow (recommended)

1. Pull latest `main`
2. Create branch
3. Work & commit
4. Push branch
5. Open PR
6. Review + merge
7. Delete branch

---

# 🧩 10. Bonus Tips (real-world useful)

* Pull often → reduces conflicts
* Keep branches small → easier reviews
* Use PR descriptions → explain your work
* Communicate → "I'm editing this file"

---

# 🧠 TL;DR

* Don’t touch `main` directly
* Use branches (`feature/...`)
* Open PRs for merging
* Conflicts happen → resolve manually
* Rebase = clean history, Merge = safer
